# Data Model: RSSRipple

## Entity Relationship Overview

RSSRipple uses a relational data model centered around **Channels** (RSS sources), **Agents** (automation logic), and **FileResources** (parsed RSS entries). The key relationships are:

```
Channel 1──N FileResource       (A channel produces many resources via RSS fetches)
Channel 1──N Agent              (A channel can have multiple agents processing it)
Agent   1──N ResourceFilter     (An agent has many filter rules)
Agent   1──1 DownloaderInstance (An agent sends tasks to one downloader)
Agent   1──N DownloadTask       (An agent creates many download tasks)
Agent   1──N PendingDecision    (An agent creates pending decisions for ambiguous matches)
FileResource N──1 Episode       (A resource may map to a specific episode)
Episode N──1 TVSeries           (Episodes belong to a TV series)
FileResource N──1 Movie         (A resource may map to a movie)
DownloadTask N──1 FileResource  (A task downloads a specific resource)
DownloadTask N──1 Agent         (A task belongs to an agent)
PendingDecision N──1 Agent      (A decision belongs to an agent)
PendingDecision N──N FileResource (A decision has candidate resources)
```

## Entities

### Channel (订阅频道)

Represents an RSS feed subscription source. Channels are periodically fetched to discover new FileResources.

| Field | Type | Nullable | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | UUID | No | auto-generated | Primary key |
| `name` | str | No | — | Human-readable channel name |
| `type` | enum | No | `rss_feed` | Channel type (currently only `rss_feed`) |
| `url` | str | No | — | RSS feed URL |
| `fetch_interval` | int | No | 1800 | Fetch interval in seconds |
| `field_mapping` | JSON | Yes | null | Dynamic field mapping rules (generated by LLM analysis) |
| `parser_type` | enum | No | `auto` | Parser type: `auto`, `mikanani`, `custom` |
| `last_fetched_at` | datetime | Yes | null | Last successful fetch timestamp |
| `status` | enum | No | `active` | Channel status: `active`, `inactive`, `error` |
| `created_at` | datetime | No | auto | Creation timestamp |
| `updated_at` | datetime | No | auto | Last update timestamp |

**Relationships:**
- Has many `FileResource` (via `channel_id`)
- Has many `Agent` (via `channel_id`)

**Constraints:**
- `url` should be unique per channel (soft constraint)
- `fetch_interval` must be positive

### ResourceFilter (资源过滤器)

Defines a single filter rule within an Agent. Filters are applied to FileResource fields to determine which resources match the Agent's preferences.

| Field | Type | Nullable | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | UUID | No | auto-generated | Primary key |
| `agent_id` | UUID | No | — | FK → Agent.id |
| `field` | enum | No | — | Target field to match. Allowed values: `subtitle_group`, `resolution`, `container`, `video_codec`, `audio_codec`, `subtitle_type`, `source`, `title_cn`, `title_en` |
| `operator` | enum | No | — | Comparison operator: `eq` (exact match), `contains` (substring), `fuzzy` (fuzzy/Levenshtein), `in` (in list), `regex` (regular expression) |
| `value` | str | No | — | The value to match against |
| `priority` | int | No | 0 | Priority weight (higher = more preferred). Used for scoring in Tier 2 |
| `is_required` | bool | No | false | If `true`, this is a hard requirement (Tier 1 filter). If `false`, it adds to the scoring (Tier 2) |
| `created_at` | datetime | No | auto | Creation timestamp |

**Relationships:**
- Belongs to `Agent` (via `agent_id`)

| `error_message` | str | Yes | null | Error description if status is `error` |
| `retry_count` | int | No | 0 | Number of retries attempted |
| `max_retries` | int | No | 3 | Maximum allowed retries |

**Status Enum:**
| Value | Description |
|-------|-------------|
| `pending` | Task created, awaiting submission to downloader |
| `queued` | Submitted to downloader, waiting in queue |
| `downloading` | Actively downloading |
| `paused` | Download paused by user or system |
| `completed` | Download finished successfully |

### PendingDecision (待决策项)

Created when the Agent cannot automatically determine a single best resource for an episode/movie. Requires LLM or human intervention.

| Field | Type | Nullable | Default | Description |
|-------|------|----------|---------|-------------|
| `id` | UUID | No | auto-generated | Primary key |
| `agent_id` | UUID | No | — | FK → Agent.id |
| `episode_id` | UUID | Yes | null | FK → Episode.id (if this decision is for a series episode) |
| `movie_id` | UUID | Yes | null | FK → Movie.id (if this decision is for a movie) |
| `candidates` | list[UUID] | No | [] | List of candidate FileResource IDs |
| `reason` | str | No | — | Human-readable reason why a decision is needed |
| `llm_suggestion` | str | Yes | null | LLM-generated recommendation text |
| `decided_resource_id` | UUID | Yes | null | The FileResource ID chosen by the user or LLM |
| `status` | enum | No | `pending` | Decision status (see Status Enum below) |
| `created_at` | datetime | No | auto | Creation timestamp |
| `decided_at` | datetime | Yes | null | Timestamp when decision was made |

**Status Enum:**
| Value | Description |
|-------|-------------|
| `pending` | Awaiting user or LLM decision |
| `decided` | A resource was chosen |
| `expired` | Decision expired (e.g., resources no longer available) |
| `skipped` | User chose to skip this decision |

**Decision Flow:**
1. Agent processes new FileResources for an episode/movie
2. After Tier 1 filtering, multiple candidates remain
3. Tier 2 (LLM) is attempted if `llm_enabled` — may produce `llm_suggestion`
4. If LLM cannot decide or is disabled, a `PendingDecision` is created with status `pending`
5. User reviews candidates and calls `POST /decisions/{id}/confirm` with a `resource_id`
6. Or user calls `POST /decisions/{id}/skip` to skip

**Relationships:**
- Belongs to `Agent` (via `agent_id`)
- May belong to `Episode` (via `episode_id`)
- May belong to `Movie` (via `movie_id`)
- References many `FileResource` (via `candidates` list)
- References one `FileResource` (via `decided_resource_id`)

## Field Mapping Rules

### Dynamic Field Mapping Structure

Each Channel can have a `field_mapping` JSON object that describes how to extract FileResource fields from RSS feed entries parsed by `feedparser`. This enables support for multiple RSS sources with different title formats and XML structures.

### Mapping Rule Format

```json
{
  "<field_name>": {
    "source": "<dotted_path>",
    "regex": "<optional_regex>",
    "group": <capture_group_index>,
    "transform": "<optional_transform>"
  }
}
```

### Rule Properties

| Property | Type | Required | Description |
|----------|------|----------|-------------|
| `source` | str | Yes | Dotted path into the feedparser entry dict. Supports array indexing. Examples: `title`, `enclosures[0].url`, `torrent.contentLength` |
| `regex` | str | No | Regular expression to apply to the source value. Uses Python `re` syntax |
| `group` | int | No | Regex capture group index (default: 0 = full match). Only used when `regex` is provided |
| `transform` | str | No | Type coercion to apply. Allowed values: `int`, `float`, `iso_datetime`, `lowercase`, `uppercase` |

### Example Field Mapping

```json
{
  "title_cn": {"source": "title", "regex": "\\]\\s*(.+?)\\s*/", "group": 1},
  "title_en": {"source": "title", "regex": "/\\s*(.+?)\\s*-", "group": 1},
  "episode": {"source": "title", "regex": "-\\s*(\\d+)\\b", "group": 1, "transform": "int"},
  "subtitle_group": {"source": "title", "regex": "^\\[(.+?)\\]", "group": 1},
  "resolution": {"source": "title", "regex": "(\\d{3,4}[pP])", "group": 1, "transform": "lowercase"},
  "video_codec": {"source": "title", "regex": "(HEVC|AVC|H264|H265|x264|x265)", "group": 1},
  "audio_codec": {"source": "title", "regex": "(AAC|FLAC|AC3|DTS)", "group": 1},
  "container": {"source": "title", "regex": "(MP4|MKV)", "group": 1, "transform": "uppercase"},
  "subtitle_type": {"source": "title", "regex": "(简繁|CHT|CHS|简日|繁日)", "group": 1},
  "torrent_url": {"source": "enclosures[0].url"},
  "file_size": {"source": "enclosures[0].length", "transform": "int"}
}
```

### Supported Target Fields

| Field | Description | Example Values |
|-------|-------------|----------------|
| `subtitle_group` | Release group name | `LoliHouse`, `Skymoon-Raws`, `ANi` |
| `title_cn` | Chinese title | `黄泉使者`, `葬送的芙莉莲` |
| `title_en` | English title | `Yomi no Tsugai`, `Frieren` |
| `episode` | Episode number | `12`, `24` |
| `resolution` | Video resolution | `1080p`, `720p` |
| `source` | Source type | `WebRip`, `WEB-DL`, `Baha`, `ViuTV` |
| `video_codec` | Video codec | `HEVC-10bit`, `AVC`, `H264` |
| `audio_codec` | Audio codec | `AAC`, `FLAC` |
| `subtitle_type` | Subtitle type | `简繁内封字幕`, `CHT`, `CHS` |
| `container` | Container format | `MP4`, `MKV` |
| `torrent_url` | Torrent file URL | `https://mikanani.me/Download/...` |
| `file_size` | File size in bytes | `718442304` |

### Parser Types

| Type | Description |
|------|-------------|
| `auto` | System auto-detects parser based on RSS source |
| `mikanani` | Uses hardcoded regex for mikanani.me title format (fallback parser in `title_parser.py`) |
| `custom` | Uses the channel's `field_mapping` JSON for parsing |

## Filter Resolution Logic

### Three-Tier Decision System

The Agent processes FileResources through a three-tier resolution pipeline:

```
Tier 1: Rule-based Filter (Required filters)
  ↓ Unique match → Download immediately
  ↓ Multiple matches → Proceed to Tier 2
  ↓ No matches → Skip (mark as unmatched)

Tier 2: Scoring + LLM Decision
  ↓ LLM selects one → Download selected
  ↓ LLM cannot decide → Proceed to Tier 3
  ↓ LLM call fails → Proceed to Tier 3

Tier 3: Human Decision
  ↓ User selects one → Download selected
  ↓ User skips → Mark as skipped
```

### Tier 1: Required Filters (Hard Constraints)

- All `ResourceFilter` entries with `is_required = true` are applied
- A FileResource must match ALL required filters to pass
- Operators are applied based on their type:
  - `eq`: exact string match (case-insensitive)
  - `contains`: substring check
  - `in`: value is in a comma-separated list
  - `regex`: regex pattern match
  - `fuzzy`: Levenshtein ratio > 0.7

### Tier 2: Optional Scoring + LLM

- All `ResourceFilter` entries with `is_required = false` are applied as scoring factors
- Each matching optional filter adds its `priority` value to the candidate's score
- The candidate with the highest total score is preferred
- If scores are tied or LLM is enabled:
  - LLM is called with candidate details and filter context
  - LLM may produce a recommendation (`llm_suggestion`)
  - If LLM confidently selects one, it is chosen
  - If LLM is uncertain or unavailable, proceed to Tier 3

### Tier 3: Human Decision

- A `PendingDecision` record is created with all remaining candidates
- The user views candidates in the UI with their parsed metadata
- User can:
  - **Confirm** a specific resource → creates a `DownloadTask`
  - **Skip** → marks the decision as `skipped`

### Consistency Matching (EpisodeProfile)

To maintain subtitle group and quality consistency across episodes of the same series:

1. When an episode is downloaded, an `EpisodeProfile` is recorded with the chosen resource's attributes:
   - `subtitle_group`, `resolution`, `container`, `video_codec`, `subtitle_type`
2. For subsequent episodes, the previous episode's profile is used as additional scoring factors
3. Matching the profile adds bonus points but is not a hard requirement

### Title Fuzzy Matching Strategy

For associating FileResources with TVSeries/Movie entities:

1. **Exact match**: Compare `title_cn` or `title_en` against series/movie titles
2. **Alias match**: Compare against `TVSeries.aliases` or `Movie.aliases`
3. **Fuzzy match**: Levenshtein ratio > 0.7 against all known titles
4. **External source**: Match via external database IDs (TMDB, Bangumi API)
